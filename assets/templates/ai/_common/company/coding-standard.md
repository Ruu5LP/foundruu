# コーディング規約

言語標準のリンター（`recommended` 相当）で機械的に拾える範囲を超えた、
人と AI が共通で守るコーディングルールを定める。
**リンターが緑でも、この規約に反する変更は不可とする。**

## 基本方針

- 読みやすさを最優先にする
- 一貫性を保つ（既存のスタイルに合わせる）
- 不要なコメントは書かない。コメントは「なぜ」を書き、「何を」はコードで伝える
- マジックナンバー・マジック文字列は名前付き定数として定義する
- 使われていないコード（変数 / 関数 / import / dead code）を残さない
- 早期 return でネストを浅く保つ（ネストは 3 段まで目安）
- 破壊的変更より不変（immutable）を優先する

## 型・null 安全（TypeScript）

基本型は厳密に扱う。**型エラーを消すためだけの「無理やり合わせるキャスト」を禁止する。**

- `any` は禁止。型が不明なら `unknown` を使い、型ガードで絞り込んでから扱う
- 次はすべて禁止（型を偽って辻褄を合わせる行為）：
  - `as any` / `as unknown as T`（二段キャスト）
  - 非 null アサーション `!`（`foo!.bar`）
  - `@ts-ignore` / `@ts-nocheck`（どうしても必要なら `@ts-expect-error` + 理由コメント）
- `as` によるキャストは最小限にし、型ガード・判別可能ユニオンを優先する
- 型のみの import は `import type` を使う
- 公開関数・エクスポート関数は引数と戻り値の型を明示する
- `null` / `undefined` は握りつぶさず、早期 return か明示的なデフォルトで処理する
- `switch` は網羅（exhaustive）にし、`default` で想定外を検知する
- ミュータブルにする理由がなければ `const` と `readonly` を優先する
- （Nuxt）`.vue` 単一ファイルコンポーネントでは型情報を使う検査（`await` 漏れ検出・`no-unsafe-*` 等）が
  リンターで効かない。**async・型が絡むロジックは composable（`.ts`）へ寄せ**、`.vue` は表示に近い層に留める

## 関数・構造

- 1 関数 = 1 責務。目安 40 行以内、超えるなら分割を検討する
- 引数は 3 つ以下を目安にし、超える場合はオブジェクト（options）にまとめる
- 副作用のある関数と純粋関数を分離する
- ブール引数でモードを分岐させない（関数を分けるか enum にする）
- ネストが深くなったら早期 return / 関数抽出で平坦化する

## 状態管理（極力状態を持たない）

- 可能な限りステートレス・純粋関数で書く。状態は「必要な場所に、必要な分だけ」持つ
- モジュールスコープの可変グローバル状態を作らない（設定は引数 / DI で渡す）
- 入力を破壊しない。引数のオブジェクト・配列を直接変更せず、新しい値を返す
- 再代入より導出（`const` + 式）を優先する
- 状態を持つならクラス / ストアなど 1 箇所に集約し、更新経路を限定する

## 非同期処理

- `await` 漏れを禁止する（浮いた Promise を放置しない）
- 独立した非同期処理は `Promise.all` で並列化する
- `async` 関数内の例外は必ずハンドリングするか、呼び出し側へ伝播させる

## 命名規則

- 省略語・1 文字変数を避ける（ループ index 等の慣例を除く）
- 意味ごとにプレフィクスを揃える（下表）。同じ役割の名前は同じ動詞・接頭辞で統一する

| 種別 | プレフィクス | 例 |
| --- | --- | --- |
| 真偽値 | `is` / `has` / `can` / `should` | `isActive`, `hasError`, `canEdit` |
| 取得（非同期含む） | `fetch` / `get` / `load` | `fetchUser`, `getConfig` |
| 生成・変換 | `create` / `build` / `to` / `parse` | `createOrder`, `toDto`, `parseDate` |
| 更新・保存 | `update` / `save` / `set` | `saveOrder`, `setName` |
| 削除 | `delete` / `remove` | `deleteUser`, `removeItem` |
| イベントハンドラ | `handle` / `on` | `handleSubmit`, `onClick` |

- 副作用のある関数は動詞始まりにする（`fetchUser`, `saveOrder`）

### TypeScript / JavaScript
- 変数・関数: `camelCase`
- クラス・型・インターフェース: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`
- ファイル名: `kebab-case.ts`

### Python
- 変数・関数: `snake_case`
- クラス: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`
- ファイル名: `snake_case.py`

### PHP（Laravel）
- 変数・メソッド: `camelCase`
- クラス: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`
- 1 ファイル 1 クラス（PSR-4）

## エラーハンドリング

- エラーは適切な型で throw/raise する（文字列 throw 禁止）
- エラーメッセージは英語で記述する
- 握り潰しは絶対に禁止（空の catch / `catch (_) {}` は NG）
- catch した例外は再 throw するか、文脈を付けてログに残す
- 例外を制御フローの代わりに使わない（想定内の分岐は戻り値で表現する）

## ファイル・ディレクトリ

- 1 ファイルが肥大化しないようにする（目安 400 行、超えたら責務で分割）
- ただし「小さすぎるファイルの乱立」も避ける。1 関数 = 1 ファイルにしない
- 1 ファイル 1 責務。関連する型・ヘルパは同じファイル / 同じディレクトリにまとめる
- ディレクトリは役割で切る（例: `domain` / `usecase` / `infra` / `ui`）。実装都合で無秩序に増やさない
- ディレクトリの深さは 3〜4 階層までを目安にする
- 詳細な構成規約は [プロジェクト規約](../project/conventions.md) を参照

## 自動チェックの範囲（機械強制と規約の境界）

この規約は言語横断で適用するが、**リンター / 静的解析で機械的に強制できる範囲は言語ごとに異なる**。
緑になったこと（自動チェックの通過）は規約遵守の十分条件ではない。機械が拾えない項目は人のレビューで担保する。

- **TypeScript / JavaScript**: ESLint（`strictTypeChecked` + 本リポジトリの追加ルール）で、
  `any` 禁止・非 null アサーション禁止・`await` 漏れ・命名（真偽値プレフィクス）・ファイル/複雑度上限などを強制する。
  `lint` は `--max-warnings 0` で実行し、警告もエラーとして扱う
- **PHP（Laravel）**: PHPStan/Larastan（最低 level 8）で型の無理な辻褄合わせを検出し、Pint で整形を統一する
- **Python**: Ruff（lint + format）と mypy（`strict` + `disallow_any_explicit`）で、
  `Any` 禁止・命名規約（`snake_case` 等）・未使用コード・型注釈の明示などを強制する。
  Ruff / mypy は指摘があれば非ゼロ終了するため、TS の `--max-warnings 0` と同等に扱う
- **機械強制されない（＝レビューで見る）例**:
  - 命名プレフィクス表のうち真偽値以外（`fetch`/`get`/`handle`/`create` など動詞の使い分け）
  - Nuxt `.vue` 内の型認識チェック（前述）
  - 関数の責務分割・副作用の分離・状態を持たない設計など、意図に関わる判断
- **上記以外の言語**: 本規約は同じく適用対象。ツールチェーンが同梱されていない言語では、
  各言語標準のリンター / 型チェッカを導入して同水準の担保を用意する

## セキュリティ

- シークレットは必ず環境変数で管理する
- `.env` をコミットしない（`.env.example` のみ）
- SQL インジェクション・XSS に注意する
- 詳細は [security.md](./security.md) を参照
