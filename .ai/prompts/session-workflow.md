# AI開発セッション ワークフロー

このドキュメントは、Claude Code / Codex などのAI開発エージェントが
1つの開発セッションを進めるためのルールです。
作業を始める前に必ずこの内容に従ってください。

---

あなたはAI開発セッションの作業者です。

最初に `requirements.md` またはユーザーから与えられた要件を確認してください。

要件が実装可能なレベルまで具体的であれば、そのまま設計・実装を開始してください。

ただし、以下に該当する場合は実装を開始してはいけません。

- 実装方法が複数考えられ、判断によって仕様が変わる
- 画面やAPIの挙動が曖昧
- DB変更の有無や内容が判断できない
- 権限・認可条件が不明
- 既存仕様への影響が大きそう
- テスト観点を作れない
- 仕様を推測しないと実装できない

この場合は `questions.md` に確認事項を書き出してください。

質問は重要度順に最大5件までにしてください。

ユーザーから回答を受けたら `requirements.md` を更新し、要件が十分になったと
判断できた時点で設計・実装へ進んでください。

要件が十分な場合は、過剰に確認せず作業を進めてください。

---

## 進め方（要件ゲート）

```
requirements.md / ユーザーの要件を読む
          │
          ▼
   要件は実装可能か？
     ├─ はい → そのまま design.md → tasks.md → 実装 → test.md → summary.md
     └─ いいえ → questions.md に確認事項（最大5件）
                       │
                       ▼
                ユーザーの回答を待つ
                       │
                       ▼
                requirements.md を更新 → 十分なら実装へ
```

- 要件確認は **最初の1回** を基本とします。
- 要件が十分なら、**ユーザー確認を挟まずに** 作業を進めてかまいません。
- 曖昧で実装方法が複数考えられる場合のみ、実装を開始せず質問します。
- 大げさなワークフロー管理にはしないでください。必要なファイルだけ埋めます。

## 各ファイルの役割

| ファイル | いつ書くか | 内容 |
| --- | --- | --- |
| `requirements.md` | 最初 | 要件。ここが判定の起点 |
| `questions.md` | 要件が不足しているときだけ | 確認事項（最大5件、重要度順） |
| `design.md` | 要件が十分になった後 | 実装方針・変更対象・DB/API/UI・影響範囲 |
| `tasks.md` | 設計後 | 実装/確認タスクと完了条件 |
| `test.md` | 設計〜実装 | テスト観点・手動/自動テスト・リグレッション |
| `summary.md` | 完了時 | 実施内容・変更ファイル・残課題・振り返り |

## 注意

- セッション成果物（`.ai/sessions/`）は Git 管理しません。作業メモとして自由に使ってください。
- テンプレートは `.ai/templates/session/` にあります。
